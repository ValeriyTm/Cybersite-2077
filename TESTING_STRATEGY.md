# План тестирования

Данный документ объединяет план ручного контроля качества и описание автоматизированных сценариев (Vitest, Cypress, Storybook).

## Инструкция по запуску тестов

- Unit- и интеграционные тесты
  Выполнить из корня проекта: "npm run test".
- E2E-тесты
  Выполнить из корня проекта: "cd apps/web && npx cypress run".

* Визуальное тестирование:
  Выполнить "npx chromatic --project-token=TOKEN", где TOKEN - это токен Chromatic, указанный в .env в apps/web/.env.

- 13.4. Теперь при каждом входе такого пользователя в аккаунт после ввода пароля должно появляться окно ввода кода из приложения на телефоне.

В течение минуты должен быть удален аккаунт из БД.

### Модуль Catalog

#### 1.Вход в каталог различными способами

- 1.1. Зайти в каталог нажатием кнопки "Каталог" компонента Header.
- 1.2. Зайти в каталог "/catalog" с главной страницы.
- 1.3. С "/catalog" перейти в "/catalog/motorcycles", а затем на страницу конкретного бренда и конкретного мотоцикла.
- 1.4. Должны работать ссылки перехода в каталог определенного бренда из выпадающего меню при наведении курсора на кнопку "Каталог" компонента Header. Ссылка "Прочие бренды" должна вести на страницу "/catalog/motorcycles.

#### 2.Страница брендов "/catalog/motorcycles"

- 2.1. Должны корректно отображаться карточки брендов (вывод количества моделей должен соответствовать реальности; изображение должно выводиться).
- 2.2. Проверить работу пагинации.
- 2.3. Проверить сохранение текущего состояния поиска и пагинации при обновлении страницы.
- 2.4. Проверить работу поиска по брендам.
- 2.5. Проверить работу компонента Breadcrumbs.

#### 3.Страница конкретного бренда "/catalog/motorcycles/{brand_name}"

- 3.1. Должны корректно отображаться карточки мотоциклов (либо выводится настоящее изображение, либо дефолтное; должны отображаться год, объем и мощность двигателя, цена и рейтинг).
- 3.2. Проверить работу пагинации.
- 3.3. Проверить работу поиска по моделям мотоциклов.
- 3.4. Проверить работу компонента Breadcrumbs.
- 3.5. Кнопки переключения режимов "Сетки" и "Списка" должны работать.
- 3.6. Проверить работу всех доступных способов сортировки (по алфавиту в обоих направлениях; по цене в обоих направлениях; по новизне; по рейтингу).
- 3.7. Проверить работу всех фильтров (цена, объем, год, мощность, категория, тип привода).
- 3.8. Проверить сохранение текущего состояния поиска, фильтров и пагинации при обновлении страницы (значения в URL должны сохраняться).

#### 4.Страница конкретного мотоцикла "/catalog/motorcycles/{brand_name}/{moto_name}"

- 4.1. Должно отображаться либо дефолтное изображение, либо реальное. Если реальных несколько, то они должны быть в виде миниатюрных изображений под основным и между ними нужно переключаться.
- 4.2. Проверить наличие имени модели, бренда, цены, года выпуска, объема двигателя и рейтинга рядом с изображением.
- 4.3. Проверить работу компонента Tabs.
- 4.4. На вкладке "Технические характеристики" данные должны отображаться реальные, либо же заглушка "Нет данных".
- 4.5. Проверить возможность скачать pdf-файл на вкладке "Документы".
- 4.6. Под блоком Tabs должны выводиться рекомендации в виде четырех карточек мотоциклов (карточки проверить по пункту 3.1.).
- 4.7. Проверить работу компонента Breadcrumbs.

### Модуль Warehouse

#### 1.Корректность работы "В наличии"

- 1.1. Проверить работу фильтра "В наличии" на странице конкретного бренда /catalog/motorcycles/{brandName}.
- 1.2. Проверить, что badge "В наличии" отображается только на моделях в наличии.
- 1.3. Кнопка добавления в корзину не должна работать для карточек (а также на странице конкретной модели), модели которых не имеют свободных остатков на складе.
- 1.4. На странице добавки в корзину кнопка создания заказа должна быть заблокирована, если товар, выбранный для заказа, имеет на складе свободных остатков меньше, чем требуется для создания заказа. Также должно выводиться предупреждение. Кнопка должна быть разблокирована при снятии выбора с проблемного товара или при изменении его количества до такого, при котором возможно создание заказа.

#### 2.Seed-скрипт

- 2.1. В БД в таблице Warehouse должно быть сгенерировано 5 складов.
- 2.2. В БД в таблице Stock должно быть сгенерировано рандомное количество (0-3) товаров для каждой позиции на каждом складе.

#### 3.Работа с картой

- 3.1. При открытии карты на ней должно стоять 5 оранжевых меток, отображающих склады, указанные в таблице Warehouse.
- 3.2. При нажатии на карту должна ставиться синяя метка. Зона установки метки должна ограничиваться исключительно территорией РФ с выводом предупреждения.
- 3.3. После установки метки должен отображаться корректный текстовый адрес для этой метки.

#### 4.Расчёт доставки

- 4.1. При указании геометки для адреса доставки должен отображаться ближайший склад, имеющий все необходимые товары в наличии. Также должно появиться отображение стоимости доставки (она должна включиться в итоговую сумму заказа) и дату доставки.
- 4.2. При последующих созданиях заказа на странице формирования заказа должен по дефолту использоваться адрес доставки прошлого заказа, на основе которого по дефолту вычисляется дата и стоимость заказа.
- 4.3. Добавить много товаров для заказа, чтобы невозможно было добиться наличия всех позиций на одном складе. При создании заказа должно появиться предупреждение, что доставка невозможна и нужно связаться с менеджером.

### Модуль Ordering

#### 1.Жизненный цикл заказа

- 1.1. После создания заказа он должен получать статус PENDING.
- 1.2. Через час после создания заказа, но без оплаты, заказ должен автоматически переводиться в статус CANCELED.
- 1.3. Для имитации оплаты использовать эндпоинт сервера "/orders/{order_id}/pay-test". После отправки patch-запроса на этот эндпоинт заказ переведется в статус PAID.
- 1.4. Через 2-3 ч после перевода в PAID статус должен автоматически переводиться в DELIVERY.
- 1.5. В расчетную дату доставки заказ автоматически должен перейти в состояние DELIVERED.
- 1.6. После нажатия кнопки подтверждения принятия товара заказ должен перейти в статус COMPLETED.
- 1.7. На странице заказов должна отображаться кнопка отмены заказа, пока тот имеет статусы PENDING и PAID. Должна отображаться кнопка подтверждения принятия заказа, когда тот имеет статус DELIVERED. Должна отображаться кнопка для оценки товара, когда заказ имеет статус COMPLETED.
- 1.8. Нажатие кнопки отмены заказа должно переводить тот в статус CANCELED.
- 1.9. Заказ должен учитываться в счетчике активных заказов в Header, если его статус - PENDING, PAID, DELIVERY.
- 1.10. Статус заказа на странице заказов должно актуализироваться автоматически каждые 30 сек.
- 1.11. Проверить работу фильтра по статусу на странице заказов.

#### 2.Жизненный цикл заказа и склад

- 2.1. (Бронирование заказа) В момент создания заказа в таблице Stock должен наполняться столбец reserved (какое количество в заказе, столько и в столбце). Для расчёта наличия товара по всему каталогу с тех пор должна использоваться разница между quantity и reserved (каждое обновление остатков должно сопровождаться соответствующей синхронизацией Elasticsearch).
- 2.2. (Возврат) При отмене заказа (пользователем или автоматчиески) столбец reserved должен очищаться.
- 2.3. (Уменьшение остатков) При оплате заказа столбец reserved должен очищаться и столбец quantity должен уменьшаться на соответствующее количество.
